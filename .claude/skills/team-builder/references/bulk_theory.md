# 耐久指数最適化の理論とアルゴリズム

このドキュメントは `pkdx hbd` サブコマンドが採用している総合耐久指数最適化の背景理論・アルゴリズム・実装上の判断を記録する。構築・育成の判断でユーザーから「なぜこの配分？」「H=B+D とは？」「11n調整は？」などと問われた時の一次資料。

## 1. 総合耐久指数の定義

- **物理耐久 HB** = H × B （HPと防御の積）
- **特殊耐久 HD** = H × D
- **総合耐久指数 HBD/(B+D)** = H·B·D / (B+D)

HB/HD は片側の耐久を表す指標。総合耐久指数は物理・特殊両方の被ダメを同時に考慮した指標で、Yakkun を含む日本コミュニティで **「正統派総合耐久指数」** として広く使われる。

## 2. なぜ HBD/(B+D) が正統派なのか

ある技から受けるダメージは概ね「攻撃力 / 防御力」に比例する。物理火力 p、特殊火力 s を想定した時の **1ターンあたり総被ダメージ** は:

```
総被ダメ ≈ p/(H·B) + s/(H·D)
       = (p·D + s·B) / (H·B·D)
       = (p·D + s·B) / HBD
```

これを最小化することは、その逆数である以下の目的関数を最大化することと等価:

```
f(H, B, D) = H·B·D / (s·B + p·D)
```

p=s=1（物理・特殊同程度に警戒）のとき、これが `HBD/(B+D)` そのもの。よって「HBD/(B+D) の最大化 ⇔ 総被ダメの最小化」が数学的に成立する。

`HB + HD`（和）は総合耐久の近似としてたまに使われるが、片方の耐久が極端に低いときにそれが無視されるため品質が劣る。`H²·B·D`（純粋な積）も候補だが H=2√(BD) を要求し HP過剰投資になる。どちらも pkdx は採用しない。

## 3. H = B + D 則の導出

EV を連続値と仮定し、H/B/D の実数値を独立変数とした Lagrange 乗数法で `HBD/(B+D)` を最大化する。偏微分を 0 にする条件から:

- 性格補正なしの理想形: **H = B + D**
- 性格補正あり（B を 1.1 倍、D を 1.0 倍など）: **H ≈ n_B·B + n_D·D** （それぞれの性格乗数で重み付け）

これはあくまで **連続近似の理想形**。実際には床関数・EV4刻み・EV252上限・EV合計508上限という離散制約が働き、ほとんどの実ポケモンで「H=B+D がちょうど成立する配分」は存在しない。pkdx が返すのは **離散制約下での実現可能な最大解**であり、連続理想形とは通常ズレる。

## 4. 連続近似の限界

pkdx `hbd` は **指標最適化** を行う。すなわち HBD/(sB+pD) を最大化する EV配分を返す。以下は別概念であり、このサブコマンドのスコープではない:

- **確定n発耐え** — 特定の技・攻撃側ステータスを前提に「N発で落ちない」条件を満たす最小 EV を逆算する計算。`calc` スキル / `pkdx damage` 側の責務。
- **想定仮想敵耐え** — 複数の具体的な攻撃側を想定した複合条件充足。個別にダメ計を回す必要がある。

「総合耐久指数が最大」は「何が飛んでくるか分からない相手に対して平均的に最も落ちにくい」であり、確定耐えとは異なる保証を提供する。

## 5. 非対称火力想定

`--phys-weight p` / `--spec-weight s` で物理/特殊被ダメの比重を指定できる。例:

- `p=2 s=1`: 環境に物理アタッカーが多く、2倍の頻度で物理攻撃を受けると想定 → 最適解は B 寄りにシフト
- `p=1 s=10`: 特殊一辺倒 → D 寄りにシフト（実質 HD の最大化に近い）

目的関数は `HBD/(s·B + p·D)` と一般化され、連続理想形は `p·D = s·B` という条件に変わる（B と D がそれぞれの重みで等しくなる点）。

## 6. アルゴリズム — greedy 勾配法

実装は lacolaco/pokemon-tools の [`optimizeDefenseEVs`](https://github.com/lacolaco/pokemon-tools/blob/main/projects/stats/src/lib/calculator/optimizer.ts) に倣った greedy 勾配法。

**偏微分**:
```
∂f/∂H = B·D / (s·B + p·D)
∂f/∂B = H·p·D² / (s·B + p·D)²
∂f/∂D = H·s·B² / (s·B + p·D)²
```

**ループ（単一解モード）**:
1. H/B/D の現実数値を求める
2. 各スタットの偏微分を計算
3. **「実数値+1 に必要な最小EV」で偏微分を割り算** → **単位EVあたりの貢献度**
4. 最大貢献度のスタットに EV を割り当てる
5. 残 EV が尽きるか全勾配0になるまで繰り返し

実数値+1 に必要な最小EVで正規化するのが肝。単に最大勾配を選ぶのではなく、**EV1ポイントあたりの寄与**で比較することで、「8EVで+2実数値が狙える場所」のような高効率境界を自動的に優先する。この正規化こそが 11n調整を暗黙に実現する機構（次節）。

**複数解モード (`--top N`, N ≥ 2)**: brute-force で (evH, evB, evD) ∈ {0,4,8,...,252}³ を sum ≤ 508-固定 で全列挙し、HP条件でフィルタしてスコア降順 + タイブレーク規則でソート、上位 N を返す。最大候補数 ~180K だが通常数万、MoonBit native で数十ms。

## 7. 11n調整との関係

11n調整は「性格+10%補正時、補正前10n／補正後11n で端数が切り捨てられない境界があり、そこで実数値効率が最大になる」という手筋（参考: https://migerug4.hatenablog.com/entry/2021/03/04/003027 ）。

本アルゴリズムは **明示的な11n制約を持たない** が、以下の2機構により自動で11n境界を捕捉する:

1. **単位EV正規化**: 11n境界で「8EVで+2実数値」のような高効率点が発生すると、その点の単位貢献度が跳ね上がり、勾配法が優先的にそこまで伸ばす
2. **低実数値優先**: 偏微分 `∂f/∂B = HpD²/(sB+pD)²` は B が小さいほど大きい。つまり「実数値の低いスタットに振る方が効率が良い」という 11n記事の主張② も自動的に反映される

結果として、migerug4 記事の論点①②③（高効率境界・低実数値優先・連続的価値）は greedy 勾配法で暗黙に実現される。**ユーザーは明示的な `--11n` オプション等なしで 11n最適解を得られる**。

## 8. HP条件一覧（`--hp-snap`）

| 値 | H 制約 | 用途・根拠 |
|---|---|---|
| `leftovers` | H ≡ 1 (mod 16) | たべのこし/くろいヘドロ回復 `floor(H/16)` を最大化しつつ奇数保持 |
| `residual` | H ≡ 15 (mod 16) | 砂嵐/毒/やけど/ステルスロック等の定数ダメ `floor(H/16)·m` の発動回数を最小化 |
| `sitrus` | H が偶数 | オボン/半減実等の 50% HP 発動しきい値に対して半分がちょうど整数となる |
| `lifeorb` | H ≡ 0 (mod 10) | いのちのたま反動 `floor(H/10)` を最小化 |
| `none` | 制約なし | （デフォルト） |

これらは効果ベースの命名であり、特定道具への対応付けはユーザー責任。例えば `residual` は砂ダメだけでなくステロや定数ダメ全般に使える。

単一解モードでは greedy ループ開始時に「H を最小の snap 有効値まで引き上げ」、以降の H 更新は常に次の snap 有効値にジャンプする。HP snap で到達不能な場合（例: ベース HP が低すぎて leftovers 条件に 252EV でも届かない）は制約を無視した解に **warning フラグ**を付けて返す。

## 9. pkdx hbd コマンド例

```bash
# デフォルト: H/B/D を最適化、A/C/S=0
bin/pkdx hbd ガブリアス --nature ようき

# S振り固定（252）で残りを H/B/D に分配
bin/pkdx hbd ガブリアス --nature ようき --fixed-ev "_,0,_,0,_,252"

# たべのこし持ち想定（16n+1）
bin/pkdx hbd ガブリアス --nature ようき --fixed-ev "_,0,_,0,_,252" --hp-snap leftovers

# 物理環境想定（被物理2倍重み）
bin/pkdx hbd カビゴン --nature ずぶとい --phys-weight 2 --spec-weight 1

# 上位5候補を比較
bin/pkdx hbd ガブリアス --nature ようき --top 5

# JSON出力（スクリプトでパースしたい時）
bin/pkdx hbd ガブリアス --nature ようき --format json
```
